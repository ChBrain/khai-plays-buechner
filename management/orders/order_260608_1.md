# Management Order: Adopt the Rider Lane Class for Orders

## Direction

An order exists to drive a change, and that change can land in any lane; most often it restages a play. Until now an order and the play it changed were forced into two pull requests, because `management/orders/**` rode the governance lane while `plays/**` rode the play lane. The khai-guard `riders` class dissolves that friction: an order rides the lane of the change it accompanies, and homes to governance only when it stands alone. The house adopts the class so an order and its execution are one unit of work, and the temporary migration lane raised for the canon upgrade is retired.

## Orders

1. The Theatre Manager (franzos) shall raise `@chbrain/khai-guard` to the version that ships the `riders` class, and `@chbrain/khai-arch` alongside it.
2. The Theatre Manager (franzos) shall declare `management/orders/**` a rider in `khai-guard.config.json`, with `governance` as its fallback lane.
3. The Theatre Manager (franzos) shall remove the temporary `migration/*` lane, now spent: an order rides its own change, so no cross-lane migration branch is needed.
4. The Theatre Manager (franzos) shall record the rider rule under Branching in `CLAUDE.md`.

## Implementation

- Raise `@chbrain/khai-guard` to `^0.1.9` and `@chbrain/khai-arch` to `^0.1.9`.
- Add `branchScope.riders` with `{ pattern: "management/orders/**", fallback: "governance" }` to `khai-guard.config.json`, and delete the `migration/*` lane.
- Record the rider rule under Branching in `CLAUDE.md`.
- Run `npm test` to verify conformance.
